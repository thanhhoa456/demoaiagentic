### 1) Summary of Requirements  
The user story describes a patient‑centric feature that enables **logged‑in patients** to **view a list of healthcare providers**, **filter and sort** that list, **view provider details** (including available appointment slots), and **book an appointment** using only **available time slots**. The system must **prevent double‑booking** of the same slot and ensure that successful bookings are confirmed and reflected in the patient’s “My Appointments” list.  

Key functional goals:  

- **Provider List** – display specialty, experience, rating, fee, location, and clinic/hospital.  
- **Filtering** – allow patients to narrow the list by provider attributes (e.g., specialty, location).  
- **Sorting** – enable sorting by check‑up fee or rating.  
- **Provider Detail Page** – show available appointment time slots.  
- **Booking** – select only slots that are currently free; upon success, confirm the appointment and add it to “My Appointments”.  
- **Concurrency Control** – ensure that when multiple patients attempt to book the same slot, only one succeeds and others receive a clear rejection message.

| **AC1** | **View Provider List** | • Patient must be **logged in** before the list is shown.<br>• System must **display** a list of **available** healthcare providers.<br>• Each provider card must contain:<br> - Medical specialty<br> - Years of experience<br> - Rating (e.g., star rating)<br> - Check‑up fee<br> - Current hospital/clinic<br> - Location (address or map link) |
| **AC2** | **Filter and Sort Providers** | • Patient can **filter** providers by any provider attribute (e.g., specialty, location).<br>• Patient can **sort** the list by:<br> - Check‑up fee (ascending/descending)<br> - Rating (ascending/descending) |
| **AC3** | **View Provider Details** | • From the provider list, patient can navigate to a **provider detail page**.<br>• The detail page must **display** all available **appointment time slots** for that provider. |
| **AC4** | **Book Appointment** | • Patient may select **only time slots that are marked as available**.<br>• Upon successful booking:<br> - Appointment is **confirmed** (e.g., status = “Booked”).<br> - Appointment appears in **“My Appointments” (Lịch của tôi)**.<br>• System must provide a **clear success notification** to the patient. |
| **AC5** | **Handle Concurrent Booking** | • System must **prevent double‑booking** of the same time slot.<br>• If multiple users attempt to book the same slot simultaneously:<br> - Only **one** booking is successfully created.<br> - All other attempts are **rejected** with an appropriate notification (e.g., “This time slot is no longer available”). |

### 3) Edge Cases and Risks  

| Category | Edge Case / Risk | Impact | Mitigation / Clarification Needed |
|----------|------------------|--------|-----------------------------------|
| **Authentication** | Patient must be **logged in**; unauthenticated access should be blocked. | Security breach, unauthorized data view. | Confirm login flow, session handling, and role‑based access control. |
| **Provider Data Completeness** | Missing or stale data (e.g., outdated years of experience, fee, location). | Wrong information displayed → user mistrust. | Define data refresh cadence; validate provider records before display. |
| **Filtering** | Filter criteria may be ambiguous (e.g., “location” could mean city, postcode, or radius). | Incomplete or incorrect list. | Clarify filter semantics; specify UI controls (dropdown, text, map). |
| **Sorting** | Sorting by fee or rating may expose business logic (e.g., hidden discounts). | Potential pricing manipulation. | Ensure sorting uses only publicly exposed fields; hide internal pricing rules. |
| **Time Slot Representation** | Time slots may be stored as **date‑time** without timezone info. | Wrong slot availability for users in different time zones. | Confirm that all times are stored in **UTC** and displayed in patient’s local timezone. |
| **Slot Availability Window** | “Available” may be defined by **business hours**, **provider schedule**, or **booking window** (e.g., 24 h in advance). | Users may try to book outside allowed windows. | Define the exact rule for slot availability; enforce server‑side. |
| **Concurrent Booking Race Condition** | Two requests may read the same “available” status before either writes, leading to **double‑booking**. | Business logic violation, revenue loss, user frustration. | Implement **optimistic locking** (version/timestamp) or **pessimistic locking** (row‑level lock) on slot table. |
| **Slot Expiration** | A slot may become unavailable after a short period (e.g., 5 min) but the UI does not reflect it instantly. | User selects a slot that later becomes booked. | Ensure real‑time status sync (WebSocket, AJAX polling) or lock slot on selection. |
| **Duplicate Providers** | Same provider may appear under multiple clinic entries → duplicate list items. | Confusing UI, wasted slots. | Define deduplication logic; enforce unique provider IDs. |
| **Large Data Sets** | Provider list may grow to thousands; filtering/sorting could become slow. | Poor performance, timeout errors. | Implement pagination, server‑side filtering, and indexing on filter/sort columns. |
| **Concurrent Edit of Provider Details** | Provider may change fee or availability while a patient is viewing the list. | Stale data displayed. | Use **optimistic concurrency** or refresh list after a short TTL. |
| **Booking Confirmation Flow** | After booking, patient may navigate away before the “My Appointments” list updates. | Perceived failure, duplicate submissions. | Ensure atomic transaction: booking and appointment list update in same DB transaction; push UI update via real‑time channel. |
| **Notification Delivery** | Failure to send success/failure notifications (email, push, in‑app). | Poor user experience, missed appointments. | Define notification channels, retry logic, and fallback UI messages. |
| **Error Messaging** | Generic “error” messages for slot unavailability may be unclear. | User confusion, support tickets. | Provide specific, user‑friendly messages (e.g., “Selected time slot is no longer available”). |
| **Data Retention / Deletion** | Deleting a patient record should not affect existing appointments (historical data). | Loss of medical history. | Define cascade rules: appointments remain, but patient view is removed. |
| **Regulatory / Compliance** | Storing patient appointment data may be subject to health‑record regulations (e.g., GDPR, HIPAA). | Legal risk. | Verify that data handling meets relevant compliance requirements. |
| **Internationalization** | Provider names, locations, fees may be in different languages/currencies. | Display errors, misinterpretation. | Ensure i18n support for UI strings, currency formatting, and locale‑aware date/time. |
| **Accessibility** | UI components (filters, sorting, slot selection) must be accessible to screen‑reader users. | Excludes users with disabilities. | Validate WCAG compliance for interactive elements. |
| **Performance under Load** | High traffic (e.g., many patients booking at the same time) may cause latency or slot contention. | Service degradation, double‑booking risk. | Conduct load testing; ensure auto‑scaling and proper caching of provider lists. |
